# Architecture

หน้านี้อธิบายโครงสร้างทางเทคนิคของ Legal ERP Platform ด้วยภาษาที่อ่านตามได้:
ใช้เทคโนโลยีอะไร หน้าจอคุยกับระบบหลังบ้านอย่างไร ข้อมูลถูกเก็บตรงไหน
ข้อมูลของแต่ละองค์กรแยกกันอย่างไร และจุดเชื่อมระบบภายนอกอยู่ตรงไหน

Architecture ปัจจุบันยึดแนวคิด web-based (ใช้งานผ่าน browser), matter-centered
(ให้ `Matter` หรือแฟ้มงานกฎหมายเป็นศูนย์กลาง) และ multi-tenant
(หลายองค์กรใช้งานระบบเดียวกัน แต่ข้อมูลต้องแยกจากกัน) โดยให้ frontend, backend
และ database (ฐานข้อมูล) แยกหน้าที่ชัดเจน แต่ทำงานบนภาษากลางชุดเดียวกัน

## Architecture Goals

- ทำให้ข้อมูล `Client` (ลูกความ/ผู้รับบริการ), `Matter` (แฟ้มงานกฎหมาย),
  `Document` (เอกสาร), `Task` (งานย่อย), `Calendar Activity`
  (นัดหมาย/กำหนดเวลา), `Quotation` (ใบเสนอราคา), `Billing`
  (การวางบิลและติดตามชำระ) และ `Finance Foundation` (ข้อมูลตั้งต้นด้านการเงิน)
  เชื่อมกันจากข้อมูลหลักชุดเดียว
- รองรับหลาย `Tenant` (องค์กรผู้ใช้งาน) โดยข้อมูลของแต่ละองค์กรต้องไม่ปนกัน
- กำหนดว่าใครดูหรือแก้ไขข้อมูลใดได้ ผ่าน `Role` (บทบาทผู้ใช้งาน), `Permission`
  (สิทธิ์การใช้งาน) และ `Audit Log` (ประวัติการเปลี่ยนแปลง)
- ต่อเติม module (ส่วนงานระบบ) ใหม่ได้ โดยไม่ทำให้ขั้นตอนหลักของ `Matter`
  ซับซ้อนเกินไป
- เตรียมจุดเชื่อมระบบภายนอกไว้ล่วงหน้า โดยยังไม่ผูกกับผู้ให้บริการรายใดรายหนึ่ง
  ในรอบเริ่มต้น

## System Context

ผู้ใช้หลักของระบบทำงานผ่าน browser (เว็บเบราว์เซอร์) โดยเข้าสู่ Legal ERP
Platform เพื่อจัดการงานประจำวันของสำนักงานกฎหมายหรือทีมกฎหมาย ระบบควรทำหน้าที่
เป็นศูนย์กลางข้อมูลระหว่างทีมกฎหมาย ทีมปฏิบัติการ ผู้ดูแลระบบ และทีมการเงิน

- **Legal Users**: ทีมกฎหมายที่ใช้งานแฟ้มงานกฎหมาย งานย่อย เอกสาร นัดหมาย
  และใบเสนอราคา
- **Finance Users**: ทีมการเงินที่ติดตามการวางบิล การรับชำระเงิน
  และข้อมูลการเงินที่เกี่ยวข้อง
- **Administrators**: ผู้ดูแลระบบที่จัดการองค์กรผู้ใช้งาน ผู้ใช้งาน บทบาท สิทธิ์
  ข้อมูลตั้งต้น และค่าตั้งต้นของระบบ
- **External Systems**: ระบบภายนอกที่อาจเชื่อมต่อในอนาคต เช่น payment gateway,
  accounting system, e-Filing, court system หรือ document signing service

## Application Layers

```mermaid
flowchart TB
  User["User via Browser<br/>ผู้ใช้ผ่านเว็บเบราว์เซอร์"]
  Frontend["Frontend Layer<br/>React + Tailwind CSS<br/>หน้าจอที่ผู้ใช้ทำงาน"]
  Backend["Backend Layer<br/>Django API / Business Logic<br/>ระบบหลังบ้าน"]
  Database["Data Layer<br/>PostgreSQL<br/>ฐานข้อมูลหลัก"]
  FileStorage["File Storage Boundary<br/>Document files / templates / exports<br/>พื้นที่ไฟล์เอกสาร"]
  Integration["Integration Boundary<br/>External systems<br/>ระบบภายนอก"]

  User --> Frontend
  Frontend --> Backend
  Backend --> Database
  Backend --> FileStorage
  Backend --> Integration
```

- **Frontend Layer (ส่วนหน้าจอ)**: React และ Tailwind CSS สำหรับสร้างหน้าจอที่
  ผู้ใช้ทำงานผ่าน browser
- **Backend Layer (ระบบหลังบ้าน)**: Django สำหรับรับคำสั่งจากหน้าจอ ตรวจสิทธิ์
  ตรวจความถูกต้องของข้อมูล จัดการขั้นตอนการทำงาน และส่งข้อมูลกลับไปยังหน้าจอ
- **Data Layer (ฐานข้อมูลหลัก)**: PostgreSQL สำหรับจัดเก็บข้อมูลหลักของระบบ เช่น
  องค์กรผู้ใช้งาน ผู้ใช้งาน ลูกความ แฟ้มงานกฎหมาย เอกสาร งานย่อย นัดหมาย
  ใบเสนอราคา ใบแจ้งหนี้ และการรับชำระเงิน
- **File Storage Boundary (พื้นที่ไฟล์เอกสาร)**: พื้นที่จัดเก็บไฟล์เอกสาร แม่แบบ
  และไฟล์ที่ส่งออก โดยเก็บข้อมูลกำกับไฟล์และสิทธิ์การเข้าถึงไว้ใน PostgreSQL
- **Integration Boundary (จุดเชื่อมระบบภายนอก)**: จุดเชื่อมกับระบบอื่น
  ควรถูกแยกชัดเจน เพื่อให้เปลี่ยนผู้ให้บริการหรือเลื่อนไป phase (ช่วงงาน)
  ถัดไปได้

## Core Backend Components

- **Tenant Management (จัดการองค์กรผู้ใช้งาน)**: จัดการข้อมูลองค์กรและค่าตั้งต้น
  ที่ทำให้ข้อมูลของแต่ละองค์กรแยกจากกัน
- **Identity and Access Control (ผู้ใช้งานและสิทธิ์)**: จัดการผู้ใช้งาน บทบาท
  สิทธิ์ และกติกาว่าใครเห็นหรือแก้ไขข้อมูลใดได้บ้าง
- **Matter Workspace (หน้ารวมงานกฎหมาย)**: ส่วนกลางที่เชื่อมลูกความ เอกสาร
  งานย่อย นัดหมาย ใบเสนอราคา การวางบิล และข้อมูลการเงินเข้ากับแฟ้มงานกฎหมาย
- **Document Service (เอกสาร)**: ดูแลแม่แบบเอกสาร เอกสารที่ระบบสร้าง
  ไฟล์ที่อัปโหลด เวอร์ชัน การอนุมัติ และประวัติการเปลี่ยนแปลงของเอกสาร
- **Quotation and Billing Service (ใบเสนอราคาและวางบิล)**: จัดการรูปแบบค่าบริการ
  ใบเสนอราคา การอนุมัติ การแก้ไข ใบแจ้งหนี้ เงื่อนไขชำระ และสถานะการรับเงิน
- **Task and Scheduling Service (งานและนัดหมาย)**: จัดการการมอบหมายงาน
  กำหนดส่งงาน นัดหมาย และการแจ้งเตือนที่ผูกกับแฟ้มงานกฎหมายหรือผู้รับผิดชอบ
- **Finance Foundation Service (ข้อมูลตั้งต้นด้านการเงิน)**: จัดเตรียมข้อมูล
  สำหรับรายการรับเงิน รายการจ่ายเงิน ใบแจ้งหนี้ การรับชำระเงิน ข้อมูลภาษี
  และรายงานการเงินเบื้องต้น
- **Audit and Reporting Service (ประวัติและรายงาน)**: บันทึกการเปลี่ยนแปลงสำคัญ
  และเตรียมข้อมูลสำหรับ dashboard (หน้าสรุปสถานะ) หรือ report (รายงาน)

## Matter-Centered Model

`Matter` (แฟ้มงานกฎหมาย) เป็นข้อมูลหลักของงานหนึ่งเรื่อง ผู้ใช้ควรเริ่มจาก
`Matter` แล้วเข้าถึงข้อมูลที่เกี่ยวข้องได้ครบถ้วน เช่น ลูกความ ประวัติการติดต่อ
เอกสาร งานย่อย นัดหมาย ใบเสนอราคา ใบแจ้งหนี้ การรับชำระเงิน และข้อมูลผลการทำงาน

ความสัมพันธ์หลักที่ควรใช้เป็นฐานสำหรับ database (ฐานข้อมูล) และ API
(ช่องทางที่หน้าจอใช้คุยกับระบบหลังบ้าน) มีดังนี้:

- หนึ่ง `Tenant` (องค์กรผู้ใช้งาน) มีผู้ใช้งาน ลูกความ แฟ้มงานกฎหมาย
  และข้อมูลตั้งต้นได้หลายรายการ
- หนึ่ง `Client` (ลูกความ/ผู้รับบริการ) มีได้หลาย `Matter` (แฟ้มงานกฎหมาย)
- หนึ่ง `Matter` เชื่อมกับเอกสาร งานย่อย นัดหมาย ใบเสนอราคา ใบแจ้งหนี้
  และการรับชำระเงินได้หลายรายการ
- หนึ่ง `User` (ผู้ใช้งาน) อาจมีบทบาทต่างกันตามองค์กรหรือหน้าที่ในแฟ้มงานกฎหมาย
- ทุกข้อมูลสำคัญควรบอกได้ว่าเป็นขององค์กรใด และมีประวัติการเปลี่ยนแปลงที่ชัดเจน

## Multi-Tenant Boundary

ทุกข้อมูลหลักต้องระบุว่าเป็นของ `Tenant` (องค์กรผู้ใช้งาน) ใด เพื่อป้องกันไม่ให้
ผู้ใช้ขององค์กรหนึ่งเห็นหรือแก้ไขข้อมูลของอีกองค์กรหนึ่งโดยไม่ตั้งใจ การออกแบบ
API, query (การดึงข้อมูล) และ permission check (การตรวจสิทธิ์)
ต้องถือว่าการแยกข้อมูลองค์กรเป็นเงื่อนไขพื้นฐานเสมอ

แนวทางรอบเริ่มต้น:

- ทุก request (คำขอจากหน้าจอ) ต้องรู้ว่าอยู่ในองค์กรใด
- การดึงข้อมูลธุรกิจต้องกรองตามองค์กรผู้ใช้งานเสมอ
- การตรวจสิทธิ์ต้องดูทั้งบทบาทระดับระบบและสิทธิ์ที่เกี่ยวข้องกับแฟ้มงานกฎหมาย
- `Audit Log` (ประวัติการเปลี่ยนแปลง) ต้องบันทึกว่าใครทำอะไร กับข้อมูลใด
  ในองค์กรใด และทำเมื่อไร

## Security and Audit

ระบบควรวางแนวทางความปลอดภัยจากการกำหนดสิทธิ์ตามบทบาทเป็นพื้นฐาน
แล้วค่อยขยายด้วยสิทธิ์เฉพาะ module (ส่วนงานระบบ) หรือสิทธิ์เฉพาะ `Matter` เมื่อ
requirement ชัดเจนขึ้น

- จำกัดการเข้าถึงข้อมูลตามบทบาทและองค์กรผู้ใช้งาน
- แยกสิทธิ์สำหรับการดู เพิ่ม แก้ไข อนุมัติ ส่งออก และลบข้อมูล
- บันทึกประวัติการเปลี่ยนแปลงสำหรับข้อมูลสำคัญ
- ป้องกันการเข้าถึงเอกสารและไฟล์ที่ส่งออกโดยไม่ผ่านการตรวจสิทธิ์
- เตรียมแนวทาง backup และ recovery สำหรับข้อมูล PostgreSQL และไฟล์เอกสาร

## Integration Strategy

รอบเริ่มต้นควรออกแบบจุดเชื่อมระบบภายนอกไว้ก่อน แต่ยังไม่ถือว่าต้องเชื่อมต่อระบบ
production ทั้งหมดตั้งแต่ phase (ช่วงงาน) แรก

ระบบที่อาจต้องรองรับในอนาคต:

- e-signature หรือ document signing service
- payment gateway หรือ bank integration
- external accounting หรือ ERP system
- court, e-Filing หรือ government service
- email, calendar หรือ notification provider

## Open Architecture Decisions

- รูปแบบ API ระหว่างหน้าจอและระบบหลังบ้านจะใช้ REST, GraphQL หรือรูปแบบอื่น
- พื้นที่เก็บไฟล์เอกสารจะใช้ local storage ใน development และ cloud object
  storage ใน production หรือไม่
- งานเบื้องหลังที่ระบบทำโดยไม่ต้องรอหน้าจอ เช่น สร้างเอกสาร ส่งออกไฟล์
  ส่งการแจ้งเตือน และเตรียมรายงาน จะใช้เครื่องมือใด
- การแยกข้อมูลองค์กรจะใช้ฐานข้อมูลร่วมพร้อม tenant key หรือแยก schema
  ตามองค์กรในอนาคต
- การเข้าสู่ระบบจะเริ่มจาก username/password ภายในระบบก่อน หรือรองรับ SSO
  ตั้งแต่ phase (ช่วงงาน) แรก
