# Release Readiness Checklist

| Document Control        | Value                          |
| ----------------------- | ------------------------------ |
| Document ID             | CHK-REL-001                    |
| Status                  | Approved                       |
| Version                 | 1.0                            |
| Process Owner           | Release Manager                |
| Approver                | Product Owner + Technical Lead |
| Control Sign-off        | Admin                          |
| Effective Date          | 14 July 2026                   |
| Last Requirement Review | 14 July 2026                   |

## Purpose

กำหนด evidence และ go/no-go gates ก่อน deploy Legal ERP release ไป environment
ที่มีผู้ใช้จริง รวมถึง rollback, communication และ post-release verification

## Release Record

ทุก release ต้องมี Release ID, version/commit, artifact digest, scope/backlog
IDs, Requirement/UAT coverage, environment, planned window, owner, approvers,
risk/exceptions, migration/rollback version และ decision timestamp

## Evidence Package

| Evidence ID | Evidence                                                                        | Owner              |
| ----------- | ------------------------------------------------------------------------------- | ------------------ |
| REL-SCP     | frozen scope, backlog IDs, dependencies and excluded gaps                       | Product Owner      |
| REL-TRC     | Requirement → Decision → SOP → implementation → UAT coverage                    | Product + QA       |
| REL-BLD     | build artifact digest, dependency/SBOM and source commit                        | Technical Lead     |
| REL-SEC     | RBAC/tenant/security tests, vulnerability and secret scan                       | Admin              |

| REL-OPS     | monitoring, alerts, runbook, capacity, backup/restore and support rota          | Operations         |

| REL-COM     | user/client/internal communication and training acknowledgement                 | Product/Operations |
| REL-DEC     | signed go/no-go decision, exceptions and residual risk                          | Release Approvers  |

## Scope and Traceability Gate

- [ ] Release scope ระบุ Backlog IDs และ Requirement IDs ชัดเจน
- [ ] Dependency ทุกตัว Done หรือมี approved stub/feature flag ที่ไม่ลด control
- [ ] Scope Gap/Discovery item ไม่ถูก deploy เป็น business feature
- [ ] Decision, SOP, status, permission และ workflow config version ถูก freeze
- [ ] Traceability ไม่มี Requirement ที่ขาด implementation/evidence/UAT
- [ ] Change หลัง freeze ผ่าน impact analysis และ re-test ที่เกี่ยวข้อง

## Build and Supply Chain Gate

- [ ] Build จาก clean source commit และ artifact digest ถูกบันทึก
- [ ] Lockfile/dependency resolution reproducible
- [ ] SAST, dependency, secret และ license scans ผ่าน policy
- [ ] Critical/High vulnerability เป็นศูนย์ หรือมี time-bound accepted exception
- [ ] Runtime config/secret ไม่ฝังใน artifact และแยก environment
- [ ] Feature flags มี owner, default, expiry และ rollback behavior

## Database and Migration Gate

- [ ] Migration ผ่าน staging clone ที่มี data volume/shape ใกล้ production
- [ ] Preflight ตรวจ duplicate, null, invalid status, cross-Tenant FK และ orphan
- [ ] Forward migration และ rollback/roll-forward rehearsal ผ่าน
- [ ] Lock time, table rewrite, index build และ downtime อยู่ใน approved window
- [ ] Backfill idempotent, resumable และมี progress/reconciliation count
- [ ] Backup snapshot และ restore test อยู่ใน RPO/RTO ที่อนุมัติ
- [ ] Application version compatibility ระหว่าง rolling deployment ถูกยืนยัน

## Security and Privacy Gate

- [ ] Tenant isolation ผ่าน API, UI, export, object storage และ audit metadata
- [ ] RBAC action/scope, self-approval และ toxic combination tests ผ่าน
- [ ] MFA/session/token revoke และ Emergency Access expiry ผ่าน
- [ ] Restricted/Privileged share/export ใช้ A1/A2 และ expiring access
- [ ] Encryption, masking, logging redaction และ secret rotation พร้อม
- [ ] Retention, Legal Hold, disposal block และ Destruction Certificate ผ่าน
- [ ] Security incident P1 contact/escalation และ evidence preservation พร้อม

## Functional and UAT Gate

- [ ] UAT cases ใน scope execute จาก DATA-UAT-001 manifest/version เดียวกัน
- [ ] P1/P2 defect เป็นศูนย์
- [ ] P3 ที่เหลือมี accepted exception, owner, due date และไม่กระทบ critical
      control
- [ ] Cross-Tenant, High-risk approval, financial posting, audit integrity และ
      Legal Hold cases Passed ทุกครั้ง
- [ ] Regression ครอบคลุม upstream/downstream ของ status/workflow/permission
      change
- [ ] Retry/idempotency, duplicate request/webhook และ fixed-clock SLA cases
      ผ่าน
- [ ] Browser/API actual result และ audit/database evidence เปิดตรวจได้

## Operations Gate

- [ ] Health/readiness probes แยก app, database, storage, queue และ integrations
- [ ] Metrics/dashboard มี request/error/latency, queue lag, job failure, SLA
      breach, auth denial/anomaly, export/share และ audit integrity
- [ ] Alert severity/contact ตรง CTRL-EXC-001 และผ่าน notification test
- [ ] Log/trace correlation มี Tenant-safe request/event IDs และไม่รั่ว
      sensitive data
- [ ] Capacity/load test ครอบคลุม report/export, document upload และ audit
      volume
- [ ] Runbook มี owner, commands/actions, evidence และ escalation สำหรับ failure
      หลัก
- [ ] Support/hypercare rota และ known issue communication พร้อม

## Rollback Readiness

| Area          | Required Plan                                                                 |
| ------------- | ----------------------------------------------------------------------------- |
| Application   | previous signed artifact, config/flag restore and session compatibility       |
| Database      | backward-compatible migration or tested roll-forward; no unsafe data deletion |
| Workflow      | pin old config version; prevent mixed authority/status interpretation         |
| Queue/Jobs    | pause/drain/replay with idempotency and no duplicate notification/posting     |
| Files/Exports | preserve object versions/checksums; revoke faulty links                       |
| Integrations  | disable adapter/webhook safely and reconcile external/internal state          |
| Communication | who declares rollback, user notice, client impact and recovery update cadence |

Rollback trigger รวม cross-Tenant exposure, unauthorized approval/payment,
material data corruption, audit integrity failure, Legal Hold bypass และ
sustained error/latency ที่เกิน approved threshold

## Deployment Sequence

1. เปิด change window, confirm backup/contacts และ freeze workflow config
2. Deploy backward-compatible schema/config
3. Deploy application/workers disabled for destructive/new scheduled actions
4. Run smoke tests: authentication, Tenant boundary, core query/command, audit
   event
5. Enable workers/integrations/feature flags ทีละชุดและตรวจ queue/idempotency
6. Run critical production-safe verification and reconciliation counts
7. Product/Technical/Admin confirm go-live or invoke rollback
8. Start hypercare and record final evidence/timestamps

## Go/No-Go Authority

| Decision Area              | Sign-off                                                                                               |
| -------------------------- | ------------------------------------------------------------------------------------------------------ |
| Scope/business readiness   | Product Owner                                                                                          |
| Build/migration/rollback   | Technical Lead + Database Owner                                                                        |
| UAT/defect status          | QA Lead                                                                                                |
| Payroll/leave readiness    | HR + Manager; Admin concurrence when Payroll/Leave is in release scope                                 |
| Security/privacy/retention | Admin                                                                                                  |
| Operations/support         | Operations Owner                                                                                       |
| Final Go                   | Product Owner + Technical Lead + Admin concurrence for every production release covered by CHK-REL-001 |

CHK-REL-001 ครอบคลุมทุก release ไป environment ที่มีผู้ใช้จริงตาม Purpose
ดังนั้น Admin concurrence เป็น mandatory sign-off ไม่มี production exemption;
หากไม่พร้อมต้อง No-Go หรือเปิด release ใหม่หลังปิดข้อกังวล

No-Go อัตโนมัติเมื่อ mandatory case ไม่ผ่าน, P1/P2 เปิด, migration/rollback
ไม่ผ่าน, artifact/evidence ระบุไม่ได้ หรือ owner สำคัญไม่พร้อมใน window ห้ามลด
severity หรือเปลี่ยน test result เพื่อให้ผ่าน gate

## Post-Release

- ภายใน 30 นาทีตรวจ health, errors, queue, login, Tenant filters และ audit
  events
- ภายใน 4 ชั่วโมง reconcile status transitions, postings, notifications และ
  exports
- ระหว่าง hypercare ทบทวน P1/P2 ทุกชั่วโมงและ P3 ตาม cadence
- ปิด release เมื่อ evidence ครบ, exceptions มี owner และ baseline/version
  register อัปเดต
- ทำ retrospective เมื่อ rollback, P1/P2 หรือ control/UAT escape เกิดขึ้น
