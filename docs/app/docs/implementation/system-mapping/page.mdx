## Purpose

กำหนด ownership boundary ระหว่าง PostgreSQL, Django API, React UI และ
audit/event สำหรับ implementation backlog ที่พร้อมพัฒนา เอกสารนี้เป็น logical
mapping ไม่ใช่ physical schema หรือ OpenAPI ฉบับสุดท้าย

## Cross-Cutting Data Rules

- Business table ทุกตารางมี id, tenant_id, created_at/by, updated_at/by และ
  version
- Foreign key ข้าม Tenant ต้องถูกป้องกันทั้ง application validation และ database
  composite constraint หรือ equivalent enforcement
- Financial/audit/version records ที่ post/finalize แล้วใช้ immutable row หรือ
  compensating/correction record ห้าม update/delete ต้นฉบับ
- Status ใช้ canonical value จาก STD-SOP-001 และ transition ผ่าน workflow
  command
- Monetary value เก็บ amount + currency + precision ห้ามใช้ floating point
- Timestamp เก็บ timezone-aware UTC instant และเก็บ source timezone สำหรับ
  calendar
- File binary อยู่ object storage; PostgreSQL เก็บ metadata, object key,
  checksum, classification, version และ retention reference
- Soft delete ใช้เฉพาะ record ที่ policy อนุญาต ข้อมูลควบคุมใช้ status lifecycle

## Integration and Event Boundary

- ใช้ provider adapter สำหรับ object storage, e-signature, email/calendar และ
  export delivery
- Webhook ทุกตัวตรวจ signature, Tenant/resource binding, replay/idempotency และ
  event time
- Business transaction เขียน outbox event ใน transaction เดียวกัน แล้ว worker
  ส่งต่อ
- Consumer เก็บ processed event ID และทำงานซ้ำได้โดยผลลัพธ์ไม่เพิ่มรายการซ้ำ
- Failed integration ไม่เปลี่ยน business record เป็น successful และต้องเปิด
  retry/exception

## Migration and Index Baseline

- Index ทุก business table เริ่มด้วย tenant_id และ query discriminator
  ที่ใช้จริง
- Partial/unique index สำหรับ active identifiers, references และ current
  versions
- Partition/retention strategy สำหรับ audit_event, access log และ notification
  history
- Migration status/permission มี preflight count, mapping table, dry run และ
  rollback
- Backfill event/evidence ต้องระบุ synthetic actor/source และห้ามปลอมเป็น user
  action
