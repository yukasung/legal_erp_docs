# Permissions

หน้านี้กำหนดแนวคิด permission และ permission matrix เริ่มต้นของ Legal ERP
Platform เพื่อใช้เป็นฐานก่อนลงรายละเอียดในแต่ละ module โดย permission เป็น
user-level authorization control แยกจาก Tenant Plan entitlement

ระบบต้องใช้ RBAC เพื่อให้ผู้ใช้งานแต่ละกลุ่มเห็นข้อมูลและฟังก์ชันตามบทบาทของ
ตนเอง พร้อม audit trail สำหรับตรวจสอบย้อนหลัง

## Authorization Gates

การอนุญาต action ต้องผ่านทุก gate ต่อไปนี้:

| Gate | Required Check | Deny Example |
| --- | --- | --- |
| Account | Active, authenticated, MFA/session policy ผ่าน | account suspended หรือ MFA ไม่ผ่าน |
| Entitlement | Tenant Plan/version เปิด Module/Feature | Tenant ไม่มี Time Tracking |
| Permission | Runtime role/bundle อนุญาต action | Lawyer ไม่มี payroll view |
| Scope | Tenant/team/Matter/own-record ตรง assignment | ผู้ใช้ข้าม Tenant หรือ Matter |
| Status | Record อยู่ในสถานะที่ action ทำได้ | แก้ Posted invoice โดยตรง |
| Authority / SoD | approval level, maker-checker และ executor ถูกต้อง | creator self-approve รายการสำคัญ |

Plan ที่สูงกว่าไม่ข้าม permission, scope หรือ SoD และ permission ที่มีอยู่ไม่เปิด
Feature ที่ Plan ไม่มี entitlement

## Permission Actions

Permission ใช้ canonical action ที่อ่านง่ายและสื่อความหมายตรงกันทั้งระบบ:

- view: ดูข้อมูลหรือรายการ
- create: สร้างข้อมูลใหม่
- edit/update: แก้ไขข้อมูลภายในสถานะและ field ที่อนุญาต
- delete/cancel: ลบข้อมูลชั่วคราว หรือยกเลิกตาม domain rule
- export: ส่งออกเอกสาร รายงาน หรือข้อมูลตาม classification/recipient control
- approve: อนุมัติรายการสำคัญตาม authority และห้าม self-approve
- assign: มอบหมายงานหรือเปลี่ยนผู้รับผิดชอบ
- manage: ตั้งค่าระบบ ผู้ใช้งาน role, permission หรือ master data

## Permission Action Mapping

คำที่ใช้บนหน้าจอและเอกสารจะเชื่อมกับ canonical permission ดังนี้:

| Display Action | Canonical Permission | Control Note |
| --- | --- | --- |
| ดู | view | ยังต้องผ่าน field masking, classification และ record scope |
| สร้าง/เพิ่ม | create | ไม่รวม approve/activate/post โดยอัตโนมัติ |
| แก้ไข | edit / update | จำกัด field/status; correction สำคัญต้องสร้าง revision |
| ลบ | delete / domain cancel | ข้อมูลสำคัญใช้ cancel, void, deactivate หรือ supersede แทน hard delete |
| อนุมัติ | approve | ต้องระบุ approval type/level/scope และ SoD |
| All | ไม่สร้างเป็น production permission เดียว | แตกเป็น explicit actions และ scope ก่อน grant |
| Export | export | ต้องผ่าน classification, recipient control และ record scope |

## Starting Rule

- ให้สิทธิ์น้อยที่สุดเท่าที่ role ต้องใช้ทำงานจริง
- Entitlement และ Permission ต้องผ่านพร้อมกัน ห้ามใช้ gate หนึ่งแทนอีก gate
- ข้อมูลของแต่ละ tenant ต้องแยกกันเสมอ
- ข้อมูลสำคัญ เช่น document, billing, payment และ admin settings ต้องมี audit
  log
- permission ที่เกี่ยวกับ approval ควรแยกจาก permission สำหรับ create/update
- การลบข้อมูลสำคัญควรถูกจำกัด และอาจใช้ soft delete หรือขั้นตอนอนุมัติ
- ตำแหน่งงานไม่ grant permission จนผ่าน approved mapping
- Feature ที่เป็น TBD ถือว่าไม่อนุมัติให้เปิด แม้ role มี permission

## Permission Matrix

## Approval Matrix

รายการที่มีผลต่อเอกสารทางกฎหมาย การเงิน หรือสิทธิ์ผู้ใช้งานควรแยกคนเตรียมข้อมูล
ออกจากคนอนุมัติ เพื่อลดความผิดพลาดและตรวจสอบย้อนหลังได้ง่าย

| Item                     | Prepare / Edit                 | Approve / Confirm                              | Audit Note                                                    |
| ------------------------ | ------------------------------ | ---------------------------------------------- | ------------------------------------------------------------- |
| Document final approval  | Lawyer, Assistant              | Manager หรือ Senior Lawyer ที่มี scoped bundle | บันทึกผู้แก้ไข ผู้ตรวจ authority/scope และเวลาที่อนุมัติ      |
| Quotation approval       | Lawyer, Finance                | Manager                                        | บันทึกส่วนลด เงื่อนไขพิเศษ และเวอร์ชัน quotation              |
| Billing adjustment       | Finance                        | Manager                                        | บันทึกเหตุผลการแก้ invoice, payment หรือยอดค้าง               |
| Payment refund/write-off | Finance                        | Manager                                        | บันทึกจำนวนเงิน เหตุผล และผู้อนุมัติ                          |
| Leave request            | พนักงาน (self-service) หรือ HR | Manager หรือ alternate HR                      | policy/balance/overlap และ requester/approver แยกกัน          |
| Payroll run              | HR                             | Manager A1; privileged/org-wide เพิ่ม Admin A2 | maker/A1/A2/System post แยกกันและ Posted immutable            |
| Personnel lifecycle      | HR หรือ scoped Manager         | alternate/scoped Manager                       | Admin execute access; System transition; reconcile ทุก domain |
| Delete / restore data    | เจ้าของข้อมูลหรือผู้รับผิดชอบ  | Manager หรือ Admin ตามประเภทข้อมูล             | ใช้ soft delete สำหรับข้อมูลสำคัญ                             |
| Permission change        | Admin                          | Manager; privileged ใช้ Manager A1 + Admin A2  | บันทึก role เดิม role ใหม่ เหตุผล scope และ expiry            |
| Sensitive export         | ผู้มีสิทธิ์ตาม module          | Manager เมื่อเป็นข้อมูลสำคัญ                   | บันทึกไฟล์ เวลา ผู้ส่งออก และขอบเขตข้อมูล                     |

## Data Boundary Rules

- Admin มีหน้าที่ตั้งค่าและช่วยแก้ปัญหาระบบ แต่ไม่ควรเห็นหรือแก้เนื้อหางาน
  กฎหมายโดยไม่จำเป็น
- Lawyer และ Assistant เห็นข้อมูลตามแฟ้มงานกฎหมาย ทีม หรือรายการที่ได้รับ
  มอบหมาย
- Finance เห็นข้อมูลที่จำเป็นต่อ quotation, billing, payment และ finance
  foundation แต่ไม่จำเป็นต้องเห็นรายละเอียดเอกสารทางกฎหมายทั้งหมด
- HR เห็นเฉพาะข้อมูลบุคลากร เงินเดือน และการลา ไม่เห็นข้อมูล Matter, document
  ทางกฎหมาย หรือ billing ของลูกค้า และข้อมูลเงินเดือนรายบุคคลไม่ควรให้ Finance
  เข้าถึงโดยตรง
- Manager เห็นภาพรวมและอนุมัติรายการสำคัญตามระดับความรับผิดชอบ ไม่ใช่เห็นทุก
  ข้อมูลของทุกทีมโดยอัตโนมัติ
- การลบ ส่งออก อนุมัติ เปลี่ยนสิทธิ์ และแก้ไขข้อมูลการเงินต้องมี audit log เสมอ
- ถ้ามีแฟ้มงานที่ต้องจำกัดเป็นพิเศษ ควรมีสิทธิ์ระดับแฟ้มงานกฎหมายเพิ่มเติมเหนือ
  role ปกติ
- Client Contact ใช้ Portal Membership และ publication scope ไม่ได้รับ internal
  role จากการมี Client record
- ตำแหน่ง เช่น Partner หรือ Owner ไม่ทำให้เห็นทุกข้อมูลจนกว่า runtime
  role/bundle และ scope ผ่าน approval

Authority, scope, maker-checker และ toxic permission combination ใช้
[Role & Permission Alignment](/docs/sops/governance/role-permission-alignment)
เป็น control หลักเมื่อข้อความสรุปในหน้านี้ไม่ละเอียดพอ

## Review Rule

ทบทวน matrix นี้พร้อม SOP-SEC-001 และ CTRL-RBAC-001 ทุกไตรมาส รวมถึงเมื่อ role,
approval authority, module scope, Plan entitlement หรือ privileged permission
เปลี่ยน
