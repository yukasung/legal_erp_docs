# Role Mapping

หน้านี้กำหนดแนวทางจับคู่ตำแหน่งและหน้าที่ของสมาชิกกับ Access Profile ในระบบ
การมีชื่อตำแหน่งไม่ได้ให้สิทธิ์เข้าถึงข้อมูลหรืออำนาจอนุมัติโดยอัตโนมัติ
ทุกการมอบหมายต้องระบุ Access Profile ขอบเขตข้อมูล และสิทธิ์รายบุคคลให้ชัดเจน

## Mapping Principles

- ใช้ Access Profile หลักที่ตรงกับงานประจำของสมาชิก
- เพิ่ม Access Profile อื่นเฉพาะเมื่อสมาชิกมีหน้าที่ในงานอีกด้านหนึ่งจริง
- สิทธิ์ Review, Approve, Export และงานที่มีความเสี่ยงสูงต้องมอบหมายแยก
- ขอบเขตต้องระบุระดับ Workspace ทีม แฟ้มงาน หรือข้อมูลของตนเอง
- ผู้จัดทำรายการสำคัญต้องไม่อนุมัติรายการเดียวกันด้วยตนเอง
- แผนของ Workspace เปิดให้ใช้โมดูล แต่ไม่ได้เพิ่ม Access Profile
  หรือสิทธิ์ให้สมาชิก

## Position to Access Profile Mapping

ตารางนี้เป็นแนวทางเริ่มต้น
ผู้อนุมัติต้องพิจารณาหน้าที่จริงและขอบเขตความรับผิดชอบ ก่อนมอบหมายสิทธิ์ทุกครั้ง

| ตำแหน่งหรือหน้าที่ | Access Profile หลัก |      Access Profile หรือสิทธิ์เพิ่มเติม       | ขอบเขตปกติ                          | ข้อจำกัดสำคัญ                                          |
| :----------------: | :-----------------: | :-------------------------------------------: | ----------------------------------- | ------------------------------------------------------ |
|       Owner        |       Manager       |   สิทธิ์ดูภาพรวมและอนุมัติตามระดับที่กำหนด    | องค์กรหรือหน่วยงานที่รับผิดชอบ      | ไม่ได้รับ Admin หรือ Finance โดยอัตโนมัติ              |
|       Admin        |        Admin        | สิทธิ์ดำเนินการด้านระบบตามคำขอที่อนุมัติแล้ว  | องค์กรที่ดูแล                       | ไม่เป็นผู้อนุมัติรายการธุรกิจที่ตนดำเนินการ            |
|      Partner       |       Lawyer        | Manager เมื่อรับผิดชอบกำกับทีมหรืออนุมัติงาน  | ทีมและแฟ้มงานที่รับผิดชอบ           | ชื่อตำแหน่งไม่ให้สิทธิ์อนุมัติทุกประเภท                |
|  Managing Partner  |       Lawyer        |   Manager และระดับอนุมัติตามที่องค์กรกำหนด    | องค์กร ทีม หรือกลุ่มงานที่รับผิดชอบ | ต้องกำหนดระดับอนุมัติแยก ห้ามอนุมานจากชื่อตำแหน่ง      |
|   Senior Lawyer    |       Lawyer        | สิทธิ์ Review หรือ Approve เอกสารตามประเภทงาน | ทีมและแฟ้มงานที่ได้รับมอบหมาย       | ห้ามอนุมัติงานที่ตนเป็นผู้จัดทำเมื่อกำหนดให้แยกคน      |
|   Junior Lawyer    |       Lawyer        |       ไม่มีสิทธิ์อนุมัติเป็นค่าเริ่มต้น       | แฟ้มงานที่ได้รับมอบหมาย             | การ Review หรือ Approve ต้องได้รับอนุมัติเพิ่ม         |
|     Accounting     |       Finance       |       สิทธิ์เฉพาะงานการเงินที่รับผิดชอบ       | ข้อมูลการเงินขององค์กร              | ไม่เห็นข้อมูลกฎหมายหรือเงินเดือนรายบุคคลเกินความจำเป็น |

## Additional User Groups

|           กลุ่มผู้ใช้งาน           | Access Profile หรือรูปแบบการเข้าถึง | ขอบเขตปกติ                          | ข้อจำกัดสำคัญ                                               |
| :--------------------------------: | :---------------------------------: | ----------------------------------- | ----------------------------------------------------------- |
| Legal Assistant / Case Coordinator |              Assistant              | ทีมและแฟ้มงานที่ได้รับมอบหมาย       | ช่วยจัดเตรียมและติดตามงาน แต่ไม่อนุมัติขั้นสุดท้าย          |
|            HR Personnel            |                 HR                  | ข้อมูลบุคลากร เงินเดือน และการลา    | แยกจาก Finance และไม่เห็นข้อมูลแฟ้มงานกฎหมาย                |
|           Client Contact           |          Portal Membership          | ข้อมูลที่เผยแพร่ให้ผู้ติดต่อรายนั้น | ไม่ได้รับ Access Profile ภายในจากการเป็นผู้ติดต่อของลูกความ |
|         Platform Operator          |      Controlled Support Access      | เฉพาะเหตุการณ์และเวลาที่อนุมัติ     | ไม่ใช้ Admin ของ Workspace และไม่มีอำนาจอนุมัติทางธุรกิจ    |

## Assignment Steps

1. ระบุตำแหน่ง หน้าที่ และงานที่สมาชิกต้องรับผิดชอบจริง
2. เลือก Access Profile หลักจาก Admin, Lawyer, Assistant, Finance, HR หรือ
   Manager
3. ระบุขอบเขต Workspace ทีม แฟ้มงาน และประเภทข้อมูลที่เข้าถึงได้
4. เพิ่ม Access Profile หรือสิทธิ์รายบุคคลเฉพาะงานที่กลุ่มหลักไม่ครอบคลุม
5. ตรวจการขัดกันของหน้าที่ โดยเฉพาะการจัดทำ อนุมัติ และดำเนินการ
6. ให้ผู้มีอำนาจอนุมัติคำขอ และให้ Admin ดำเนินการตามผลอนุมัติ
7. บันทึกเหตุผล ผู้ขอ ผู้อนุมัติ ผู้ดำเนินการ ขอบเขต และวันหมดอายุ
8. ทดสอบสิทธิ์หลังมอบหมายและทบทวนสิทธิ์ตามรอบ

## Common Examples

| กรณี                                       | การจับคู่ที่เหมาะสม                               | เหตุผล                                              |
| ------------------------------------------ | ------------------------------------------------- | --------------------------------------------------- |
| Partner ดูแลแฟ้มงานและอนุมัติเอกสารของทีม  | Lawyer + Manager พร้อมขอบเขตทีม                   | แยกงานกฎหมายออกจากอำนาจกำกับงาน                     |
| Senior Lawyer ตรวจร่างเอกสารเฉพาะบางประเภท | Lawyer + สิทธิ์ Review/Approve ตามประเภทเอกสาร    | ไม่ให้สิทธิ์อนุมัติครอบคลุมทุกแฟ้มงาน               |
| Admin เพิ่มสิทธิ์ตามคำขอ                   | Admin เป็นผู้ดำเนินการ และ Manager เป็นผู้อนุมัติ | ป้องกันผู้ดำเนินการอนุมัติคำขอของตนเอง              |
| Accounting จัดทำ Invoice และ Payment       | Finance ภายในขอบเขตการเงิน                        | ไม่ต้องเพิ่ม Lawyer หรือสิทธิ์ดูเอกสารกฎหมายทั้งหมด |
| HR จัดทำ Payroll                           | HR และ Manager อนุมัติรอบเงินเดือน                | แยกข้อมูลบุคลากรออกจากงาน Finance ของลูกความ        |

## Review Triggers

ต้องทบทวน Access Profile และสิทธิ์เมื่อเกิดเหตุการณ์ต่อไปนี้:

- เริ่มงาน ย้ายทีม เปลี่ยนตำแหน่ง หรือสิ้นสุดการทำงาน
- เปลี่ยนผู้รับผิดชอบทีมหรือแฟ้มงาน
- เพิ่มหรือลดอำนาจอนุมัติ
- เปิดใช้โมดูลหรือฟีเจอร์ใหม่
- พบสิทธิ์เกินหน้าที่ การอนุมัติตนเอง หรือการเข้าถึงข้อมูลผิดขอบเขต
- ถึงรอบทบทวนสิทธิ์หรือวันหมดอายุของสิทธิ์ชั่วคราว
