## Purpose

กำหนดวิธีสร้าง เปลี่ยน ระงับ และปิดบัญชี รวมถึงมอบ role/permission ตามหลัก least
privilege, tenant isolation และ separation of duties

## Scope

ครอบคลุม Tenant user, invitation, activation, role assignment, individual
permission, privileged access, password/MFA, suspension/deactivation, ownership
transfer, Emergency Access, recertification และ anomaly/access review

## Roles

| Role    | Responsibility                                                    |
| ------- | ----------------------------------------------------------------- |
| Manager | ยืนยันความจำเป็น ขอบเขต และระยะเวลาของ access และเป็น A1          |
| Admin   | เตรียม/ดำเนินการ account, role และ permission change; เป็น A2 ได้ |
| User    | ยืนยันตัวตน ใช้ MFA และรักษา credential                           |
| Auditor | ตรวจ request, approval, recertification และ audit events          |

Admin ผู้ดำเนินการห้ามอนุมัติคำขอของตนเอง และสิทธิ์ไม่ทำให้ผู้ใช้เห็นข้อมูลข้าม
Tenant หรือข้าม Matter restriction

## Account Status

| Status         | Meaning                                          |
| -------------- | ------------------------------------------------ |
| Pending Invite | สร้างบัญชีแล้วแต่ยังไม่ยืนยันตัวตน               |
| Active         | ผ่าน activation/MFA และใช้งานตาม permission      |
| Locked         | ระบบล็อกจาก authentication/anomaly condition     |
| Suspended      | ระงับชั่วคราวโดยยังเก็บ assignment/history       |
| Inactive       | ยุติการใช้งานและ revoke session/integration แล้ว |

## Access Authority

| Action                                       | Requester  | Approval                 | Executor |
| -------------------------------------------- | ---------- | ------------------------ | -------- |
| Create standard user                         | Manager    | Manager/Tenant Authority | Admin    |
| Assign standard role                         | Manager    | Manager/Tenant Authority | Admin    |
| Individual permission                        | Manager    | Manager                  | Admin    |
| Admin/audit/export/finance-approve privilege | Manager    | Manager A1 + Admin A2    | Admin    |
| Suspend/deactivate                           | Manager    | Manager                  | Admin    |
| Emergency Access                             | Admin/User | Authorized Manager       | Admin    |

Requester ห้ามอนุมัติคำขอของตนเอง Individual/Emergency Access ต้องมี expires at
และถูกถอนอัตโนมัติเมื่อครบกำหนด

Admin A2 ต้องเป็นคนละบัญชีกับ Admin ผู้ execute คำขอเดียวกัน

## Procedure

### 1. Create and Activate a User

**Responsible:** Manager (Requester), Admin (Executor)

1. ตรวจ Tenant, package/user limit และ email uniqueness
2. บันทึกชื่อ email, employment/engagement status, team, manager และ start date
3. เลือก baseline role ตามหน้าที่จริงและระบุ access scope
4. รับ approval ตาม Access Authority
5. ส่ง invitation แบบ expiring token และบังคับ password policy/MFA
6. Activate เมื่อยืนยัน identity, MFA และ terms ครบ
7. บันทึก request, approval, role/permission snapshot และ activated at

**Expected result:** Active user มีสิทธิ์ขั้นต่ำและอยู่ใน Tenant ที่ถูกต้อง

### 2. Change Role or Permission

**Responsible:** Manager (Requester), Admin (Executor)

1. ระบุ business reason, current/new role, resource scope, effective/expiry date
2. ตรวจ SoD conflict, Matter restriction และ package limit
3. Standard change ใช้ Manager approval; privileged change ใช้ A1/A2
4. ให้ Admin ดำเนินการหลัง approval มีผลเท่านั้น
5. Revoke สิทธิ์เดิมที่ไม่จำเป็นและ invalidate cached authorization
6. แจ้ง User/owner และสร้าง immutable permission-change event

**Expected result:** Permission snapshot ใหม่ตรงกับหน้าที่และไม่มีสิทธิ์ตกค้าง

### 3. Suspend or Deactivate a User

**Responsible:** Manager, Admin

1. ระบุ reason, effective time และ leaver/suspension type
2. Revoke active session, token, API key, calendar/email integration และ MFA
3. โอน Matter, Task, Appointment, approval queue และ report ownership
4. ระงับ account ก่อน ownership transfer เมื่อมี security/compliance risk
5. เก็บ user/role/history ห้าม hard delete
6. ตรวจ orphaned assignment และบันทึก reconciliation

**Expected result:** ผู้ใช้ไม่เข้าระบบได้และงานมี owner ใหม่โดยประวัติยังครบ

### 4. Grant Emergency Access

**Responsible:** Requester, Authorized Manager, Admin

1. ระบุ incident/purpose, scope, requested permissions และ expires at
2. รับ approval ก่อน grant เว้นแต่ break-glass policy ที่อนุมัติไว้กำหนด
3. บังคับ Acting Role และแสดง banner/notification ว่าใช้ Emergency Access
4. บันทึกทุก view/change/export ระหว่าง session
5. ถอนอัตโนมัติเมื่อครบเวลาและ revoke sessions
6. Manager/Admin review activity และ outcome หลังใช้งาน

Emergency Access ใช้แก้ระบบ/การเข้าถึงเท่านั้น ห้ามอนุมัติ business decision

**Expected result:** Access ชั่วคราวมีขอบเขต เวลา และ post-use review

### 5. Recertify Access

**Responsible:** Manager, Admin

1. ทบทวน privileged role และ individual permission ทุกไตรมาส
2. ทบทวน user/role ทั้งหมดอย่างน้อยปีละครั้ง
3. ตรวจ inactive user, dormant account, SoD conflict และ expired assignment
4. Manager ยืนยัน Keep/Modify/Revoke พร้อม reason
5. Admin ยืนยัน privileged access และ exception
6. Admin ดำเนินการและเก็บ before/after snapshot

**Expected result:** สิทธิ์ได้รับการยืนยันตามรอบและรายการเกินจำเป็นถูกถอน

### 6. Review Access Anomalies

**Responsible:** Admin

1. ตรวจ failed login, impossible travel, unusual export และ privilege change
2. Triage severity, affected Tenant/User/records และ evidence
3. Lock/Suspend account เมื่อเข้า incident threshold
4. แจ้ง Manager และเปิด incident record
5. Restore access หลัง identity/risk validation และ approval

**Expected result:** Anomaly มี disposition และ response audit trail

## Control Points

| Control             | Requirement                                               | Evidence                                     |
| ------------------- | --------------------------------------------------------- | -------------------------------------------- |
| Tenant isolation    | User/role/resource ต้องอยู่ Tenant เดียวกัน               | Tenant validation และ authorization decision |
| Least privilege     | ให้เฉพาะ role/action/scope ที่จำเป็น                      | Access request และ role snapshot             |
| Maker-checker       | Admin executor แยกจาก approver                            | Requester/approver/executor IDs              |
| Privileged approval | Admin/audit/export/finance approve ใช้ A1/A2              | Dual approvals และ SoD result                |
| Authentication      | Password policy, MFA, session/token revocation            | Authentication/MFA events                    |
| Time-bound access   | Individual/Emergency Access ต้องมี expiry และ auto-revoke | Grant/revoke events                          |
| Ownership transfer  | Leaver ต้องไม่มี orphaned operational record              | Reconciliation result                        |
| Immutable audit     | Login, access, permission, export และ emergency activity  | Event chain และ integrity verification       |

## Exceptions

- **User limit เต็ม:** ห้ามสร้าง/activate จนแก้ package หรือ deactivate user
- **Email ซ้ำ:** ใช้บัญชีเดิมหรือแก้ identity ห้ามสร้างซ้ำ
- **SoD conflict:** ลด permission หรือใช้ alternate approver
- **Leaver เร่งด่วน:** Suspend/revoke ก่อน แล้วทำ ownership reconciliation
- **MFA ใช้ไม่ได้:** ใช้ approved recovery flow ห้ามปิด MFA ถาวรเพื่อแก้ชั่วคราว
- **Emergency Access เกินเวลา:** auto-revoke และเปิด exception/incident
- **Anomaly ร้ายแรง:** Lock/Suspend โดยไม่รอรอบ recertification

## Completion Checklist

- [ ] Tenant, identity, email และ user limit ผ่าน validation
- [ ] Role/permission มี business reason, scope และ approval
- [ ] Privileged access มี A1/A2 และ SoD
- [ ] MFA, password/session/token controls ทำงาน
- [ ] Individual/Emergency Access มี expiry และ revoke event
- [ ] Leaver ownership reconciliation ไม่มีรายการกำพร้า
- [ ] Quarterly/annual recertification ถูกบันทึก
- [ ] Access/anomaly/audit events ตรวจย้อนหลังได้
