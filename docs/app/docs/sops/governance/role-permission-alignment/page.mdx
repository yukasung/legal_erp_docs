## Purpose

ยืนยันว่า role และ permission baseline ตรงกับ authority ใน SOP พร้อมกำหนด scope,
maker-checker และข้อห้ามสำหรับสิทธิ์สำคัญ

## Approved Role Baseline

| Role      | Primary Scope                                           | Explicit Boundary                                                  |
| --------- | ------------------------------------------------------- | ------------------------------------------------------------------ |
| Admin     | Tenant configuration, users, roles, security, execution | ไม่อนุมัติ business decision หรือเปิดดู legal content โดยไม่มีเหตุ |
| Lawyer    | Assigned Matter, Client, Document, Task                 | ไม่อนุมัติสิทธิ์ การเงิน หรือ Matter ของตนเองเมื่อ SOP ให้แยกคน    |
| Assistant | Assigned operational support                            | ไม่อนุมัติเอกสารขั้นสุดท้าย การเงิน หรือ access privilege          |
| Finance   | Quotation, invoice, payment and finance records         | ไม่ต้องเห็น legal content เกิน billing need                        |
| HR        | Personnel, payroll and leave                            | ไม่เห็น Matter, legal document หรือ client billing                 |
| Manager   | Scoped oversight, assignment and business approval      | ไม่ได้สิทธิ์เห็นทุก Tenant/ทีมโดยอัตโนมัติ                         |

Senior Lawyer ใช้ Lawyer + permission bundle สำหรับ review/approve ไม่สร้าง role
หลักเพิ่มใน baseline นี้ Admin หมายถึง Tenant Admin ส่วน platform operator
ต้องใช้ service/support access ที่แยกและมี Emergency Access control

## Responsibility and Authority Mapping

ชื่อด้านล่างเป็น responsibility assignment หรือ permission bundle ไม่ใช่
baseline role ใหม่ และต้องผูกกับผู้ใช้ที่มี baseline role พร้อม scope ชัดเจน
ข้อสรุปสุดท้าย (ไม่มี TBD หรือประเด็นเปิดค้างแล้ว): authority ทั้งสองรายการ เป็น
assignment บน baseline role ไม่ใช่ role ใหม่ โดย High-risk A2 ใช้ **Manager A2**
และ security/privacy/retention/payroll control A2 ใช้ **Admin A2** ผู้อนุมัติ A2
ต้องเป็นคนละคนกับ A1 และ Admin A2 ต้องเป็นคนละบัญชีกับ Admin executor ของ
รายการเดียวกัน

## SoD Rules

- Requester, approver และ executor ต้องเก็บเป็นคนละ audit field
  แม้บางกรณีบุคคลซ้ำได้ตาม matrix
- ผู้สร้างห้ามเป็นผู้อนุมัติรายการ High risk, privileged access, personnel
  lifecycle/sensitive correction, refund/write-off หรือ disposal ของตนเอง
- Admin เป็น executor ด้านระบบ ห้ามแทน Business Approver
- Acting Role ต้องแสดง role ที่ใช้ในธุรกรรมและเก็บ role ปกติของผู้ใช้ไว้ด้วย
- Emergency Access ห้ามใช้อนุมัติ business decision และต้องมี post-use review
- Permission aggregation ต้องตรวจ toxic combination เช่น create + approve +
  execute ในรายการเดียวกัน

## Access Review Checklist

- [ ] Role สอดคล้องกับหน้าที่และ employment/engagement status
- [ ] Tenant, team, Matter และ record scope ถูกจำกัด
- [ ] Approval permission แยกจาก prepare/execute ตาม matrix
- [ ] Individual access มี expiry; privileged access มี A1/A2
- [ ] Dormant, leaver และ transferred user ไม่มีสิทธิ์ตกค้าง
- [ ] Toxic combination และ self-approval ถูกป้องกัน
- [ ] Before/after permission snapshot และ revoke event ตรวจย้อนหลังได้
