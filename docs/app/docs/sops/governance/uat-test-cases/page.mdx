## Purpose

กำหนด UAT ที่พิสูจน์ว่า requirement, workflow, permission, audit และ exception
ใน SOP ทำงานครบก่อน release โดยทุกผลทดสอบต้อง map กลับ Requirement ID ได้

## Entry Criteria

- Build และ release scope ถูกระบุและ deployment สำเร็จใน UAT environment
- Requirement, SOP, role/permission และ test data version ถูก freeze
  สำหรับรอบทดสอบ
- มี Tenant แยกอย่างน้อยสองรายเพื่อทดสอบ isolation
- มีผู้ใช้ Admin, Lawyer, Assistant, Finance, HR, Manager และ dual approver ตาม
  scope
- Integration stub/sandbox, clock/timezone และ notification channel พร้อม
- Test data ไม่มีข้อมูลจริงที่ไม่ได้รับอนุญาต

## Execution Record

แต่ละ case ต้องบันทึก Test Run ID, build, environment, tester, role, Tenant,
test data IDs, executed at, expected/actual result, status, defect/exception ID
และ evidence ตาม FRM-UAT-001

## Standard Test Procedure

ทุก case ใช้ขั้นตอนซ้ำได้ดังนี้ โดย Scenario ในตารางเป็น action/input variation
ของขั้นตอนที่ 3:

1. Reset test data ตาม profile และบันทึก before snapshot/reference
2. Sign in ด้วย Actor role ใน Tenant A และยืนยัน permission/assignment
   prerequisite
3. ทำ Scenario ด้วย input IDs/values ที่บันทึกใน FRM-UAT-001 ตามลำดับ UI/API
   จริง
4. ตรวจ status, validation, authority, calculation, notification และ side
   effects
5. Query audit/evidence reference และตรวจ
   actor/action/object/time/outcome/Tenant
6. เปรียบเทียบ Actual กับ Expected Result และบันทึก Passed/Failed/Blocked
7. สำหรับ negative/isolation case ตรวจว่า source record ไม่เปลี่ยนและ Tenant B
   ไม่เปิดเผยทั้ง content และ metadata

Evidence package ใช้ ID รูปแบบ UAT-BUILD-CASE-RUN และต้องมี input snapshot,
screen/API result, record IDs และ audit event reference ไม่ใช้ screenshot
อย่างเดียว

แบบ FX-CASE-ID เช่น CLI-03 ใช้ STP-CLI-REG + FX-CLI-03 ห้าม tester คิดค่าใหม่

คงเดิมและต้อง export seed manifest/checksum แนบ FRM-UAT-001

## Result Status

| Status  | Meaning                                                    |
| ------- | ---------------------------------------------------------- |
| Not Run | ยังไม่เริ่ม                                                |
| Passed  | Actual ตรง expected และ evidence ครบ                       |
| Failed  | Actual ไม่ตรง expected หรือ control/evidence ขาด           |
| Blocked | ทดสอบไม่ได้เพราะ dependency/environment พร้อม defect/owner |

ห้ามใช้ Passed เมื่อมีเพียง screenshot แต่ไม่มี audit/record reference ที่ case
กำหนด และห้ามเปลี่ยน Failed เป็น Passed โดยไม่ rerun หลังแก้ไข

## Exit Criteria

- UAT cases ใน release scope ถูก execute ครบและ map Requirement ID ได้
- P1/P2 defect เป็นศูนย์ และ P3 ที่เหลือมี accepted exception/owner/due date
- Security, cross-Tenant, financial posting, Legal Hold และ audit integrity
  cases ผ่านทั้งหมด
- Regression ของ workflow ที่ได้รับผลกระทบผ่าน
- Evidence เปิดอ่านได้และ Test Run/Build/Data ID ครบ
- QA Lead สรุปผล และ Business/Process Owner ยืนยัน release readiness
