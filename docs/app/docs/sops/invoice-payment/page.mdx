## Purpose

กำหนดวิธีออก Invoice รับและจัดสรร Payment จัดการ Deposit/Receipt และแก้รายการ
ทางการเงินด้วย adjustment ที่อนุมัติ โดยยอดและประวัติตรวจสอบย้อนหลังได้

## Scope

ครอบคลุม Sales Order/accepted commercial source, Deposit, Invoice, Payment,
Receipt, outstanding receivable, allocation, reconciliation, void, refund,
credit adjustment และ write-off

## Roles

| Role    | Responsibility                                                |
| ------- | ------------------------------------------------------------- |
| Finance | เตรียม/validate/post Invoice, Payment, allocation และ receipt |
| Lawyer  | ยืนยัน scope/milestone และดู billing ของ Matter ที่รับผิดชอบ  |
| Manager | อนุมัติ exception, void, refund, write-off และ adjustment     |
| Admin   | ตั้ง tax, numbering, payment method และ finance master data   |

Finance ผู้จัดทำ adjustment/refund/write-off ห้ามเป็นผู้อนุมัติรายการเดียวกัน

## Status Baseline

### Invoice

Draft, Posted, Partially Paid, Paid, Overdue, Voided และ Written Off

### Payment

Received, Unallocated, Partially Allocated, Allocated, Refunded และ Reversed

Posted Invoice, allocated Payment และ issued Receipt ห้ามแก้หรือลบย้อนหลัง

## Required Information

### Invoice

- Invoice ID/number, Tenant, Client, Matter และ commercial source ID
- Invoice date, due date, currency, exchange-rate source/date
- Service/milestone lines, quantity/rate, tax, discount, deposit deduction
- Subtotal, tax, total, outstanding และ status

### Payment

- Payment ID, payer, source/method, bank/reference, received date และ evidence
- Amount, currency, fee, exchange rate และ received by
- Invoice allocations, unallocated balance และ reconciliation status

### Adjustment

- Record type, original record ID, amount, reason, evidence, requester, approver
  และ effective date

## Procedure

### 1. Prepare and Post an Invoice

**Responsible:** Finance

1. เลือก approved Sales Order, Accepted Quotation, Signed Engagement หรือ
   approved billing milestone
2. ตรวจ Client, Matter, billing contact, tax ID และ currency
3. ระบุ line items, tax, payment terms, due date และ deposit deduction
4. ตรวจยอด invoiced-to-date ไม่เกิน approved source โดยไม่มี Manager exception
5. Validate numbering, calculation, duplicate invoice และ required fields
6. Post Invoice และสร้าง immutable posting event
7. ส่ง Invoice/notification ตาม permission และเก็บ delivery history

**Expected result:** Posted Invoice มียอดและ commercial linkage ที่ถูกต้อง

### 2. Record a Deposit

**Responsible:** Finance

1. ตรวจ deposit term จาก accepted commercial source
2. บันทึก amount, source, received date, payer, method และ evidence
3. แยก Deposit liability ออกจาก revenue ตาม finance policy
4. Allocate กับ Client/Matter และออก receipt เมื่อ validation ผ่าน
5. ใช้ Deposit กับ Invoice โดยเก็บ utilization event และยอดคงเหลือ

**Expected result:** Deposit receipt/utilization reconcile กับ source และ
Invoice

### 3. Record and Allocate a Payment

**Responsible:** Finance

1. ตรวจ payer, bank/reference, amount, currency, date และ payment evidence
2. ค้น duplicate payment จาก reference, amount, date และ payer
3. บันทึก Received/Unallocated ก่อน allocation
4. Allocate กับ Invoice ที่ถูกต้องโดยยอดรวมไม่เกิน available amount
5. อัปเดต Invoice เป็น Partially Paid/Paid ตามยอดจริง
6. ออก Receipt หลัง allocation ผ่าน validation
7. บันทึกผู้รับ ผู้จัดสรร และ timestamps

**Expected result:** Payment, allocation, Invoice balance และ Receipt ตรงกัน

### 4. Reconcile and Monitor Outstanding Balance

**Responsible:** Finance

1. Reconcile Invoice, Deposit, Payment, Receipt และ bank/source record
2. ตรวจ unallocated payment, unmatched reference และ variance
3. ระบุ Overdue เมื่อพ้น due date และยังมียอดค้าง
4. แจ้ง Finance owner, Responsible Lawyer และ Manager ตาม collection policy
5. บันทึก follow-up, promise-to-pay และ next action
6. ห้ามเปลี่ยนยอดเพื่อปิด variance โดยไม่มี approved adjustment

**Expected result:** Outstanding/variance มี owner และ reconciliation evidence

### 5. Adjust, Void or Reverse

**Responsible:** Finance (Requester), Manager (Approver)

1. ระบุ original record, adjustment type, amount, reason และ evidence
2. ตรวจ tax, Client/Matter, downstream receipt/report และ period status
3. รับ Manager approval ก่อน credit adjustment, posted void หรือ reversal
4. สร้าง adjustment/credit/reversal record ใหม่ ห้ามแก้ original record
5. Recalculate balance และ reconcile affected records
6. แจ้ง owner/Client เมื่อ policy หรือกฎหมายกำหนด

**Expected result:** การแก้ไขโปร่งใสและยอดหลัง adjustment reconcile

### 6. Refund or Write Off

**Responsible:** Finance (Requester), Manager (Approver)

1. ตรวจ refundable/uncollectible amount และ original payment/invoice
2. ระบุ beneficiary, method, reason, tax impact และ supporting evidence
3. แยก requester, approver และ executor
4. รับ Manager approval และดำเนินการตาม approved amount
5. บันทึก transaction reference, effective date และ reconciliation
6. อัปเดต status เป็น Refunded/Written Off โดยคง original history

**Expected result:** Refund/write-off มี authority, evidence และยอดที่ reconcile

## Control Points

| Control              | Requirement                                                  | Evidence                             |
| -------------------- | ------------------------------------------------------------ | ------------------------------------ |
| Valid source         | Invoice อ้างอิง approved commercial source/milestone         | Source ID/version และ validation     |
| Calculation          | Line, tax, deposit, allocation และ balance ต้อง reconcile    | Calculation/reconciliation result    |
| Duplicate prevention | ตรวจ invoice number และ payment reference/amount/date/payer  | Duplicate-check event                |
| Posted immutability  | ห้ามแก้/ลบ Posted, Allocated หรือ Receipt เดิม               | Original/new linkage และ event chain |
| Adjustment authority | Manager อนุมัติ void, reversal, refund, write-off            | Request, approval และ SoD result     |
| Period control       | Closed period แก้ด้วย current-period adjustment เท่านั้น     | Period ID และ posting date           |
| Access/export        | Finance เห็นตามหน้าที่ Lawyer เห็นเฉพาะ Matter ที่เกี่ยวข้อง | RBAC decision และ export event       |
| Retention            | ใช้ระยะเวลาที่ยาวกว่าระหว่าง 10 ปี กฎหมาย/ภาษี/สัญญา         | Policy version และ Legal Hold        |

## Exceptions

- **Commercial source ไม่ครบ:** ห้าม Post Invoice
- **ยอดเกิน approved source:** ขอ Manager exception หรือแก้ commercial source
- **Duplicate payment:** ระงับ allocation และให้ Finance review
- **Payment ไม่ทราบ Invoice:** คง Unallocated พร้อม owner/review date
- **Closed accounting period:** ใช้ adjustment ใน period ปัจจุบัน ห้าม reopen
  โดยไม่มี authority
- **Refund account ไม่ตรง:** หยุดและตรวจ beneficiary/anti-fraud control
- **Reconciliation ไม่ลงตัว:** คง exception เปิดและห้ามออก final report
- **มี Legal Hold:** ห้าม disposal แม้ transaction ปิดแล้ว

## Completion Checklist

- [ ] Invoice ใช้ approved source และ numbering ไม่ซ้ำ
- [ ] Tax, currency, deposit และ total calculation ถูกต้อง
- [ ] Payment ผ่าน duplicate check และมี evidence
- [ ] Allocation/Receipt/Invoice balance reconcile
- [ ] Outstanding/variance มี owner และ next action
- [ ] Adjustment/refund/write-off มี Manager approval และ SoD
- [ ] Original posted records ไม่ถูกแก้หรือลบ
- [ ] Audit, period และ retention controls ถูกบันทึก
