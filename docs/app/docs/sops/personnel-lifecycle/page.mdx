## Purpose

กำหนดวิธีสร้าง ดูแล แก้ไข และยุติข้อมูลพนักงานตลอดวงจรการจ้างงาน
โดยรักษาความถูกต้องของ organization assignment, manager, employment status,
ข้อมูลส่วนบุคคลที่อ่อนไหว และหลักฐานการอนุมัติ พร้อมส่ง effective event ไปยัง
Access, Task/Calendar, Payroll/Leave และ Retention โดยไม่ให้ HR เป็นผู้อนุมัติ
และผู้ดำเนินการรายการสำคัญของตนเอง

## Scope

ครอบคลุม Employee Master, organization/team/manager assignment, employment type
และ start/end date, joiner, mover, leave of absence, suspension, leaver,
correction, self-service data request, privacy classification, Legal Hold และ
retention ไม่ครอบคลุมการคำนวณเงินเดือน/ยอดลา ซึ่งอยู่ใน SOP-HR-001 หรือการ
grant/revoke บัญชีและสิทธิ์โดยตรง ซึ่งอยู่ใน SOP-SEC-001

## Roles

| Role     | Responsibility                                                                                                     |
| -------- | ------------------------------------------------------------------------------------------------------------------ |
| HR       | จัดทำ Employee Master, ตรวจหลักฐาน, ขอแก้ไข/ย้าย/เปลี่ยนสถานะ และ reconcile ข้อมูลบุคลากร                          |
| Manager  | ยืนยันตำแหน่ง ทีม ผู้บังคับบัญชา วันที่มีผล และอนุมัติ mover/leaver ของทีม โดยห้ามอนุมัติรายการที่ตนเป็น requester |
| Admin    | ดำเนินการบัญชี สิทธิ์ และ configuration หลัง business approval; ไม่เป็น business approver ของรายการเดียวกัน        |
| Employee | ดูข้อมูลของตนเองใน field ที่อนุญาตและส่ง correction request; แก้ sensitive/master field โดยตรงไม่ได้               |
| System   | ตรวจ uniqueness/effective date, apply transition, publish event และบันทึก immutable evidence                       |

Finance ไม่มีสิทธิ์ดู Employee Master, compensation, bank, tax หรือ statutory
identifier รายบุคคล การส่งยอด payroll แบบ aggregate ใช้ SOP-HR-001

## Approved Status Baseline

Draft, Preboarding, Active, On Leave, Suspended, Terminated และ Archived

- Draft เปลี่ยนเป็น Preboarding เมื่อ required identity/employment fields ครบ
- Preboarding เปลี่ยนเป็น Active เมื่อถึง start time และ access readiness ผ่าน
- Active เปลี่ยนเป็น On Leave/Suspended/Terminated ได้เมื่อมี approved effective
  event เท่านั้น
- Terminated ห้ามกลับเป็น Active โดยแก้ record เดิม; rehire ต้องสร้าง employment
  episode ใหม่และอ้างอิง Employee ID เดิม
- Archived เป็นการจำกัดการใช้งานหลัง retention review ไม่ใช่การ hard delete

## Required Information

- Employee ID, Tenant, linked user/membership และ employment episode
- Legal/display name, employment type, legal entity, organization/team,
  position, manager, work location และ start/end/effective time
- Status, reason, requester, approver, executor, expected version และ
  idempotency key
- Sensitive field classification สำหรับ national identifier, date of birth,
  address, bank, tax, statutory และ attachment
- Before/after snapshot, ownership/access/payroll/leave reconciliation result,
  retention class และ Legal Hold result

## Procedure

### 1. Register a Joiner

**Responsible:** HR (preparer), Manager (approver), Admin/System (executor)

1. HR ตรวจ duplicate Employee ID และ identity binding ภายใน Tenant
2. สร้าง Draft พร้อม required employment fields และเก็บ sensitive fields ใน
   encrypted HR domain ที่ mask ใน UI/log
3. Manager ยืนยัน position, team, manager, start time และ access profile ที่ขอ
4. ระบบเปลี่ยนเป็น Preboarding และส่ง approved joiner event ให้ SOP-SEC-001
5. Admin ดำเนินการ account/membership/access ตาม approved profile; System
   activate เมื่อถึง start time และ access readiness ผ่าน

**Expected result:** Employee Active เพียงครั้งเดียว มี identity/employment
binding และหลักฐาน prepare/approve/execute ครบ

### 2. Change Assignment or Employment Data

**Responsible:** HR, current/new Manager, Admin/System

1. HR สร้าง mover/correction request พร้อม effective time, reason และ
   before/after
2. current Manager และ new Manager ยืนยัน ownership/assignment เมื่อย้ายข้ามทีม
3. Manager ที่มี authority อนุมัติ sensitive employment correction โดยเป็นคนละ
   บัญชีกับ HR maker
4. System สร้าง revision ใหม่ ห้ามเขียนทับ revision ที่มีผลแล้ว
5. ส่ง event ให้ Access, Task/Calendar และ Matter ownership ทำงานตาม effective
   time แล้ว reconcile role/scope, open work และ notification

**Expected result:** assignment ใหม่มีผลตามเวลา ไม่มีสิทธิ์หรือเจ้าของงานตกค้าง
และ revision เดิมยังตรวจสอบได้

### 3. Place Employee On Leave or Suspend Employment

**Responsible:** HR (requester), Manager (approver), Admin/System (executor)

1. อ้างอิง approved leave/employment event และกำหนด period/reason
2. Manager อนุมัติ restriction ที่ต้องใช้ระหว่าง On Leave/Suspended
3. System เปลี่ยน status ตาม effective time และส่ง access restriction request
4. เมื่อครบกำหนด ให้คืน Active เฉพาะเมื่อ HR ยืนยัน employment readiness และ
   Access reconciliation ผ่าน

**Expected result:** status และ access restriction สอดคล้องกันโดยไม่ทำลาย
employment history

### 4. Process a Leaver

**Responsible:** HR หรือ Manager (requester), Manager (approver), Admin/System

1. เปิด leaver request ระบุ last working day, effective termination time, reason
   class และ successor
2. ตรวจ open Matter, task, calendar, document ownership, active access,
   payroll/leave finalization และ Legal Hold
3. Manager ที่ไม่ใช่ requester อนุมัติ termination และ ownership plan
4. เมื่อถึง effective time System เปลี่ยนเป็น Terminated และส่งคำสั่งให้ Admin
   revoke session/access ตาม SOP-SEC-001
5. transfer ownership, ปิด payroll/leave balance ตาม SOP-HR-001 และบันทึก
   reconciliation; ห้าม hard delete Employee Master

**Expected result:** การจ้างสิ้นสุดตรงเวลา ไม่มี active access/งานไร้เจ้าของ
และข้อมูลยังคงอยู่ตาม retention/Legal Hold

### 5. Review, Correct and Retain Employee Data

**Responsible:** Employee/HR (request), Manager (approve), System/Admin
(execute)

1. Employee ส่ง correction request สำหรับ own permitted fields; HR ตรวจหลักฐาน
2. sensitive/master correction ต้องมี Manager approval และสร้าง revision ใหม่
3. ระบบบันทึก field classification, accessor, purpose, before/after hash และ
   decision โดยไม่เขียน sensitive value ลง general audit log
4. หลัง Terminated ให้จำกัดการเข้าถึงตาม need-to-know และคำนวณ retention trigger
5. การ archive/dispose ต้องผ่าน SOP-AUD-001; Legal Hold block disposal และ retry
   เดิมต้องคืนผลเดิม

**Expected result:** ข้อมูลถูกต้อง ตรวจย้อนหลังได้ และไม่ถูกลบก่อนครบ authority

## Access and Privacy Rules

| Actor    | Permitted View / Action                                                                  | Prohibited                                                                  |
| -------- | ---------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Employee | own non-sensitive profile; submit correction                                             | edit master/sensitive field; view other employee                            |
| Manager  | direct-report operational profile, assignment/status                                     | compensation, bank, tax, statutory or unrelated team                        |
| HR       | Tenant-scoped personnel data needed for HR purpose; prepare lifecycle/correction request | self-approve sensitive correction, mover or termination prepared by same HR |
| Admin    | approved execution metadata needed for access/config                                     | browse HR values or act as business approver and executor                   |
| Finance  | no personnel record                                                                      | all individual personnel/payroll values                                     |

## Control Points

| Control                     | Requirement                                                                     | Evidence                                                   |
| --------------------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| Unique identity             | Employee ID unique ต่อ Tenant และ user binding ไม่ซ้ำใน active episode          | uniqueness/binding result                                  |
| Maker-checker-executor      | requester/maker, business approver และ Admin/System executor แยก actor          | request, decision, execution events                        |
| Effective lifecycle         | status/assignment/access เปลี่ยนตาม approved effective time และ idempotency key | transition and downstream delivery/retry events            |
| Sensitive field boundary    | field-level authorization, masking, encryption และ purpose-based access         | policy decision/access event without raw sensitive value   |
| Revision immutability       | correction/mover สร้าง version ใหม่พร้อม before/after hash                      | linked employee revision                                   |
| Cross-domain reconciliation | Access, ownership, task/calendar และ payroll/leave ต้องครบก่อนปิด event         | reconciliation checklist and exception reference           |
| Retention and Legal Hold    | termination ไม่ลบข้อมูล; disposal ผ่าน schedule, approval และ hold check        | retention assignment, hold result and disposal certificate |

## Exceptions

- **Duplicate Employee ID/user binding:** block และส่ง HR review ห้าม merge
  employment records อัตโนมัติ
- **Start date มาถึงแต่ access ไม่พร้อม:** คง Preboarding, เปิด exception
  และห้าม ให้สิทธิ์ผ่าน manual bypass
- **Manager ไม่พร้อมอนุมัติ:** route ไป alternate Manager ตาม authority matrix;
  HR maker ห้าม approve แทนรายการของตนเอง
- **Urgent termination:** Suspend session/access ตาม emergency authority ก่อน
  แล้วต้อง complete leaver approval/reconciliation ภายใน exception SLA
- **Downstream delivery fail:** status event คงเดิม, retry ด้วย idempotency key
  และเปิด exception; ห้ามสร้าง transition ซ้ำ
- **Legal Hold:** Terminated/Archived ได้แต่ dispose ไม่ได้จน Hold ถูก release
