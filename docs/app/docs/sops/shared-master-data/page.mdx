## Purpose

กำหนดวิธีสร้าง ตรวจสอบ อนุมัติ เผยแพร่ เปลี่ยน และยุติ Shared Master Data ให้ทุก
Tenant ใช้คำอ้างอิงเดียวกันโดยไม่เกิดข้อมูลซ้ำ ไม่เขียนทับค่าที่ธุรกรรม
เดิมอ้างอิง และสามารถตรวจย้อนกลับ version/effective date ได้

## Scope

ครอบคลุม Court, jurisdiction, police station, case code, Matter/case type และ
shared legal/administrative reference data รวมทั้ง Platform Shared baseline,
Tenant Extension, duplicate search, hierarchy/linkage, approval, versioning,
effective scheduling, activation/deactivation, import/export, dependency impact,
correction และ rollback

ไม่ครอบคลุม finance/tax/rate/budget master ใน SOP-FIN-001, payroll master ใน
SOP-HR-001, quotation catalog/pricing ใน SOP-BIL-001 หรือ user/role/permission
ใน SOP-SEC-001 การเพิ่ม Tenant Extension ไม่อนุญาตให้แก้ canonical Platform
Shared record

## Roles

| Role                  | Responsibility                                                                                     |
| --------------------- | -------------------------------------------------------------------------------------------------- |
| Requester             | เสนอ record/change พร้อม source, purpose, scope และวันที่ต้องการใช้                                |
| Tenant Admin          | ขอ Tenant Extension และตรวจผลกระทบเฉพาะ Tenant; ไม่แก้ Platform Shared โดยตรง                      |
| Data Steward          | ตรวจ canonical identity, duplicate, hierarchy, source, quality และเตรียม exact version             |
| Manager A1            | อนุมัติ business/legal meaning และผลกระทบต่อ workflow/report                                       |
| Admin A2              | อนุมัติ platform-wide/high-risk change และแยกจากผู้ execute                                        |
| Platform Admin/System | schedule/activate/deactivate/import/rollback หลัง approval และบันทึก immutable events              |
| Domain Owner          | ยืนยัน mapping/dependency ของ Matter, Document, Billing, Finance, HR หรือ Report ที่ใช้ master นี้ |

## Approved Status Baseline

Draft, Pending Approval, Scheduled, Active, Inactive, Superseded และ Rejected

- Draft/Pending Approval ห้ามใช้ใน business transaction
- Scheduled ต้องมี approved version และ effective time
- Active เท่านั้นที่เลือกใช้กับรายการใหม่ได้
- Inactive ห้ามเลือกใหม่ แต่ transaction เดิมยังแสดง snapshot/reference เดิม
- Superseded ต้องชี้ replacement/version ใหม่และห้ามลบประวัติเดิม
- Rejected เก็บ reason/evidence และแก้เป็นรายการใหม่หรือ revision

## Required Information

- Master Request ID, master type, Platform Shared/Tenant Extension scope และ
  Tenant ID เมื่อเกี่ยวข้อง
- canonical code/name, aliases, local/English name, parent/hierarchy, status และ
  replacement ID
- Court/jurisdiction/police-station location/authority หรือ case type/code
  classification ตามชนิดข้อมูล
- authoritative source, source date/version, evidence checksum และ reviewer
- effective from/to, requested reason, affected domains, records/workflows และ
  migration/communication plan
- requester, Data Steward, Manager A1, Admin A2, executor, exact version,
  idempotency key และ timestamps
- duplicate candidates, validation result, before/after, import batch/control
  totals และ rollback/reconciliation result

## Procedure

### 1. Submit Master Data Request

**Responsible:** Requester, Tenant Admin, Domain Owner

1. เลือก master type และ scope เป็น Platform Shared หรือ Tenant Extension
2. ระบุ canonical code/name, aliases, hierarchy, source, purpose และ effective
   date
3. แนบ authoritative evidence โดยไม่ใส่ sensitive content เกินจำเป็น
4. ระบุ domain/workflow/report ที่คาดว่าจะได้รับผล
5. ระบบสร้าง immutable Request ID และคง record เป็น Draft

**Expected result:** request ระบุ identity, source, scope และผลกระทบเบื้องต้นครบ

### 2. Validate Identity, Duplicate and Hierarchy

**Responsible:** Data Steward, System

1. normalize code/name/alias/location ตาม master type
2. ค้นหา exact/near duplicate ทั้ง Active, Scheduled, Inactive และ Superseded
   ภายใน scope ที่เกี่ยวข้อง
3. ตรวจ parent/jurisdiction/hierarchy ว่ามีอยู่ Active และไม่เกิด circular link
4. ตรวจ code uniqueness, required fields, source date/version และ effective
   period ไม่ซ้อนกับ version เดียวกัน
5. หากเป็น Tenant Extension ให้ตรวจว่าไม่ shadow/override canonical Platform
   Shared ID โดยไม่ผ่าน approved mapping

**Expected result:** มี duplicate/hierarchy/source validation ที่ทำซ้ำได้

### 3. Assess Dependency and Change Impact

**Responsible:** Data Steward, Domain Owner

1. ดึงจำนวน Matter, Document, Task, Billing, Finance, HR, integration และ Report
   ที่อ้าง record/version เดิม
2. แยกผลต่อรายการใหม่ รายการที่กำลังทำ และประวัติที่ปิดแล้ว
3. กำหนด mapping/replacement, migration rule, effective time, notification และ
   rollback plan
4. high-risk/platform-wide change ต้องระบุ control totals และ UAT evidence
5. ห้ามเสนอ hard delete เมื่อมี dependency หรือ audit/retention obligation

**Expected result:** approver เห็น before/after, population และ recovery path

### 4. Review and Approve Exact Version

**Responsible:** Data Steward, Manager A1, Admin A2

1. Data Steward submit exact version เป็น Pending Approval
2. Manager A1 ตรวจ meaning, source, duplicate, hierarchy และ business impact
3. Platform-wide/high-risk, merge/replacement หรือ bulk import ต้องได้ Admin A2
   คนละบัญชีกับ requester/executor
4. approver อนุมัติหรือ reject exact version พร้อม reason
5. การแก้หลังอนุมัติต้องสร้าง revision และอนุมัติใหม่

**Expected result:** Scheduled version มี authority และ evidence ครบ

### 5. Schedule and Activate

**Responsible:** Platform Admin/System

1. System ตรวจ approval, expected version, idempotency key และ effective time
2. activate record/version เมื่อถึงเวลาที่อนุมัติ; retry คืนผลเดิม
3. เมื่อแทน version เดิม ให้เปลี่ยนเดิมเป็น Superseded/Inactive ตาม decision
   โดยไม่แก้ reference/snapshot ของธุรกรรมย้อนหลัง
4. publish cache/search/API event และตรวจผลในแต่ละ dependent domain
5. บันทึก before/after, actor, version และ activation result

**Expected result:** รายการใหม่เลือก version ที่ Active และประวัติเดิมคงเดิม

### 6. Correct, Deactivate or Replace

**Responsible:** Data Steward, Manager A1, Admin A2, System

1. typo ที่ยังไม่ถูกใช้แก้ผ่าน revision ได้; record ที่ถูกใช้แล้วต้องสร้าง
   replacement/version ใหม่
2. ระบุ effective end, replacement ID และผลต่อ open workflow
3. approve ตาม risk/scope เดียวกับการสร้าง
4. Inactive/Superseded ห้ามใช้กับรายการใหม่ แต่ยังค้น/รายงานย้อนหลังได้
5. merge ใช้ surviving canonical ID และ mapping event ห้ามเปลี่ยน historical ID

**Expected result:** correction ไม่ทำลาย reference หรือ audit history

### 7. Import or Export Master Data

**Responsible:** Data Steward, Admin/System

1. ใช้ approved template/schema/version และ reserved batch/idempotency key
2. validate ทุก row ก่อน commit พร้อม duplicate/hierarchy/error report
3. bulk import แบบ platform-wide ต้อง A1/A2 และใช้ all-or-controlled-partial
   rule ที่ระบุล่วงหน้า
4. reconcile input/accepted/rejected/created/updated counts และ checksum
5. export เก็บ purpose, filters, scope, recipient, as-of และ checksum ตาม
   SOP-REP-001

**Expected result:** batch ตรวจยอดและ replay ได้โดยไม่สร้าง record ซ้ำ

### 8. Roll Back Failed Activation

**Responsible:** Data Steward, Admin/System, Manager A1 + Admin A2

1. หยุดการเลือกใช้ version ที่ผิดและเปิด exception ตาม severity
2. ห้ามลบ record/version/event ที่ activate แล้ว
3. activate approved previous/compensating version หรือ mapping ตาม rollback
   plan
4. reconcile dependent records, cache/search/API/report และแจ้ง Domain Owner
5. ปิด exception เมื่อ control totals ตรงและมี prevention action

**Expected result:** บริการกลับสู่ version ที่อนุมัติโดยหลักฐานไม่ขาด

## Control Points

| Control                      | Requirement                                                               | Evidence                                                    |
| ---------------------------- | ------------------------------------------------------------------------- | ----------------------------------------------------------- |
| Canonical identity           | code/ID unique, alias และ scope ชัดเจน                                    | normalized values and duplicate candidates                  |
| Source and stewardship       | มี authoritative source/version และ Data Steward review                   | source checksum and review event                            |
| Version/effective control    | transaction ใหม่ใช้ Active version ตาม effective time                     | version/effective snapshot and activation event             |
| Actor separation             | requester/steward, A1/A2 และ executor แยกตาม risk                         | request, decisions and execution IDs                        |
| Dependency preservation      | referenced record ห้ามแก้/ลบย้อนหลัง; replacement มี mapping              | dependency population and original/replacement links        |
| Tenant boundary              | Tenant Extension ไม่ override/cross Tenant หรือแก้ Platform Shared โดยตรง | scope validation and authorization event                    |
| Bulk/import reconciliation   | row counts, checksum และ retry/idempotency ตรง                            | batch manifest and accepted/rejected/control totals         |
| Rollback without destruction | ใช้ compensating/previous version และคง original events                   | exception, rollback decision and post-change reconciliation |

## Exceptions

- **พบ duplicate:** block activation และให้ Data Steward เลือก reuse, alias,
  merge request หรือ reject; ห้ามสร้างต่อเพราะชื่อสะกดต่างเล็กน้อย
- **source ขัดแย้ง/หมดอายุ:** คง Draft/Pending และขอ Domain Owner/authority
  ตัดสิน ห้ามเดาค่า
- **hierarchy/circular link ผิด:** block ทั้ง request/batch ก่อน publish
- **มี dependency สูง:** schedule version/replacement และ migration plan;
  ห้ามแก้ Active record ตรง ๆ
- **Tenant Extension ชน canonical:** ใช้ mapping/alias หรือขอ Platform change
  ตาม authority
- **partial import fail:** ใช้กติกา batch ที่อนุมัติ, reconcile rejected rows
  และ ห้าม retry โดยไม่มี idempotency key
- **activation กระทบ workflow/report:** freeze version ใหม่ เปิด exception และ
  rollback ด้วย approved version โดยคง original event
